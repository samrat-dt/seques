Honest state of things

Where we are today

Phase 1 is complete and running. Here are the decisions we made and the trade-offs we accepted.

→

We chose reliability over raw speed at beta scale

Parallel processing at concurrency=10 was 6-8× faster but produced unpredictable answer ordering under Groq rate limits. We shipped with ANSWER_CONCURRENCY=1 by default — sequential, predictable, every question answered. Speed is a Phase 2 optimisation after async LLM clients and a Redis rate-limit tracker are in place.

→

Draft-first answers mean you always get something to work with

The engine doesn't say "I can't answer this." It produces a professional draft for every question — assertive when docs support it, hedged when they don't. Hedged answers include a note on what to verify. Trade-off: hedged answers need your review before sending to the prospect.

→

We ingest up to 32KB of your compliance docs per request

That covers most SOC 2 summaries and policy docs. Long-form reports (100KB+) get truncated. Phase 2 adds RAG with vector search — the engine will retrieve only the relevant sections per question, not the whole document.

→

Sessions are in-memory — no auth, no persistence across restarts

Supabase schema is ready and the CRUD layer is scaffolded. We chose not to wire it in Phase 1 because the in-memory store was fast to ship and the beta audience is small. A server restart clears sessions. Phase 2 persistence and auth changes this.

→

Three LLM providers available; Groq is default

The engine supports Groq (default), Anthropic, and Google via a single LLM_PROVIDER env var switch. Groq is free-tier and fast. Anthropic produces the best structured JSON. No vendor lock-in — swapping takes one env var change and a restart.

Phase 2 roadmap

What's coming next

Auth, persistence, and smarter document handling. In that order.

⚡

Supabase Magic Link auth

Email-based auth with no passwords. You enter your email, get a link, click it, and you're in. Sessions will be scoped to your account with Supabase row-level security — no one else can read your answers. Beta access will be invite-only through this flow.

⚡

Full session persistence

Sessions, questions, and answers will be stored in Supabase — surviving server restarts and available across devices. Work on a questionnaire, close the tab, come back a week later. Everything's there.

⚡

RAG for large compliance document sets

Vector embeddings for compliance docs, stored in Supabase pgvector. Instead of sending your entire SOC 2 report to the LLM, the engine retrieves the top-k relevant sections per question. Better answers, lower cost, no 32KB ceiling.

⚡

Per-user guardrails and multi-tenant isolation

Max active sessions per user, max questions per session, rate limits by user ID (not just IP). RLS policies ensure complete data isolation between teams. Ready for multi-tenant deploy once auth lands.

⚡

True parallel processing with async LLM clients

Phase 2 revisits the concurrency design with async LLM clients and a Redis-backed rate-limit budget tracker. At ANSWER_CONCURRENCY=10 with async safety, the 15-20s wall time becomes the default — not the exception.
